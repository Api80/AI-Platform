# ADR-0001：采用 Platform → Domain → System → Module 分层

- 状态：Accepted
- 日期：2026-07-13

## 背景

项目最初从新闻获取系统出发，随后逐步加入事件提取、知识图谱、因子、量化模型、回测、决策和解释能力。

如果把所有能力都视为一个“大系统”，会导致边界不清、职责混乱，也会让平台化目标变成过度设计。

需要明确通用能力、行业能力、具体系统和内部模块之间的层次。

## 决策

采用以下分层：

```text
Platform
  -> Domain
    -> System
      -> Module
```

- Platform：提供跨行业可复用的基础能力。
- Domain：封装具体行业语义和知识。
- System：领域内职责相对独立的业务系统。
- Module：系统内部具体实现单元。

TruePulse 被定义为金融 Domain 的第一个落地应用，而不是 AI Platform 本身。

新闻系统、行情系统、因子系统、量化系统等属于 TruePulse 内的并列 System。

## 理由

1. 避免把新闻系统误认为整个平台。
2. 通用能力可以被多个领域复用。
3. 金融领域内部的系统边界更清晰。
4. 支持从模块化单体逐步演进到微服务。
5. 允许先验证 TruePulse，再抽象真正通用的平台能力。

## 影响

### 正面影响

- 文档和代码可以按统一层次组织。
- TruePulse 与未来其他领域不会混在一起。
- 每个系统可以独立演进。
- 平台化目标更清晰。

### 风险

- 过早抽象可能增加复杂度。
- 如果没有真实复用需求，通用平台可能成为空架子。

## 约束

平台化是架构方向，不代表第一阶段必须实现全部平台能力。

第一阶段以 TruePulse 的真实需求为主，只有经过验证且确实可复用的能力，才逐步下沉到 Platform Layer。
