# Personal Knowledge Assistant

## 1. 定位

Personal Knowledge Assistant（个人知识助理）是 AI Platform 面向个人场景的应用方向。

它不是普通笔记工具，也不只是一个知识库，而是一个长期演进的个人信息整理与检索入口。它负责帮助用户持续收集、整理、理解和调用个人信息资产，并逐步扩展到记忆辅助、关系上下文、消息过滤和决策支持。

TruePulse 是 AI Platform 的金融领域应用；Personal Knowledge Assistant 是个人领域应用。二者共享平台能力，但业务目标、数据结构和使用场景不同。

---

## 2. 需要解决的问题

当前个人知识管理的核心问题不是“没有内容”，而是：

1. 信息入口过多：短视频、公众号、网页、聊天、截图、文档等内容分散在不同平台。
2. 收藏不等于沉淀：收藏后的内容往往没有再次阅读，也没有形成可复用知识。
3. 记忆容易丢失：曾经看过某个观点或资料，但真正需要时无法找到来源。
4. 分类成本过高：人工整理、打标签、归档容易中断，最终形成新的信息负担。
5. AI 无法直接理解个人历史：没有稳定、结构化、可检索的个人知识资产，AI 助理就无法真正长期协作。

因此，这个应用的第一目标不是建立复杂系统，而是建立一个稳定闭环：

```text
统一入口
  ↓
内容解析
  ↓
AI 摘要与分类
  ↓
人工快速确认
  ↓
长期存储
  ↓
自然语言检索与调用
```

## 3. 核心原则

### 3.1 统一入口优先

用户看到有价值的链接、文字、截图、语音或文件时，只需要把内容发送到一个固定入口。

用户不应在采集阶段思考复杂分类。采集动作必须足够简单，否则系统无法长期使用。

### 3.2 AI 负责初步整理，人负责最终确认

AI 自动完成：

- 提取标题和来源
- 生成摘要
- 推荐分类
- 自动打标签
- 判断是否值得长期保存
- 提取关键人物、组织、主题和事件
- 推荐后续行动

用户只需要确认、修改或删除。

### 3.3 原始内容必须保留

AI 摘要不是原始资料的替代品。

系统必须同时保存：

- 原始内容或原始链接
- 抓取时间
- 来源平台
- AI 生成摘要
- 分类与标签
- 内容版本
- AI 处理过程和模型信息

这是后续重新分析、模型升级和纠错的基础。

### 3.4 数据长期可迁移

核心内容优先使用开放格式保存，例如 Markdown、JSON 和标准数据库。

不把个人长期记忆完全锁定在某个特定软件、平台或模型中。

### 3.5 先沉淀，再智能化

第一阶段的重点是把内容集中、保存和整理，而不是立即建设复杂的向量库、知识图谱或自主 Agent。

复杂能力应在数据积累和真实需求出现后逐步增加。

### 3.6 隐私和权限默认严格

个人知识库未来可能包含关系、经历、聊天和决策信息，因此系统必须从一开始考虑：

- 本地优先或私有化存储
- 数据加密
- 细粒度访问控制
- 敏感数据分级
- 操作审计
- 数据导出与删除
- AI 可访问范围控制

## 4. 与其他知识体系的区别

### 4.1 与 GitHub 项目知识库的区别

当前 `AI-Platform` GitHub 仓库用于沉淀：

- 项目方法论
- 平台架构
- ADR
- 设计原则
- 研究结论
- 路线图

它是项目的 Single Source of Truth。

Personal Knowledge Assistant 面向个人生活和长期信息资产，内容来源更杂、隐私等级更高，不应该直接把全部个人内容存进公开 GitHub 仓库。

### 4.2 与 TruePulse 的区别

TruePulse 处理金融领域的数据、新闻、因子、模型、策略和回测。

Personal Knowledge Assistant 处理个人信息、知识、关系上下文和日常决策。

两者可以复用以下 AI Platform 能力：

- 数据采集
- 内容解析
- 实体识别
- 分类与标签
- 向量检索
- 知识图谱
- Agent 调度
- 可解释输出
- 权限控制

但领域模型和数据隔离必须独立。

## 5. 第一阶段：个人知识收集与整理

第一阶段目标：解决“看过、收藏过，但找不到”的问题。

### 5.1 输入类型

- 网页链接
- 公众号文章
- 短视频链接
- 纯文字
- 图片与截图
- PDF 和文档
- 语音转写
- 聊天片段

### 5.2 最小数据结构

每条知识记录至少包含：

| 字段 | 说明 |
|---|---|
| id | 唯一标识 |
| title | 标题 |
| source_type | 来源类型 |
| source_url | 原始链接 |
| raw_content | 原始内容或引用 |
| summary | AI 摘要 |
| category | 主分类 |
| tags | 标签 |
| entities | 人物、组织、产品、地点等实体 |
| created_at | 收集时间 |
| processed_at | AI 处理时间 |
| status | 待处理、已确认、已归档、已删除 |
| confidence | AI 整理置信度 |
| provenance | 内容和结论的来源链路 |

### 5.3 初期建议技术形态

第一版可以采用：

- Obsidian：个人浏览、编辑和双向链接
- NAS：原始文件和知识库备份
- Markdown：主要知识格式
- 一个轻量 API：接收链接、文字、截图和文件
- AI API：摘要、分类、标签和实体提取
- SQLite 或 PostgreSQL：保存结构化索引和处理状态

此阶段不强制引入向量库。

---

## 6. 第二阶段：智能检索与 RAG

当内容积累到一定规模，传统文件名和标签检索不足时，再增加：

- 文本切分
- Embedding
- 向量库
- 语义检索
- RAG 问答
- 引用来源返回
- 多条件混合检索

技术候选：

- PostgreSQL + pgvector
- Qdrant
- Milvus
- Elasticsearch/OpenSearch 混合检索

原则：向量库是索引，不是事实来源。原始 Markdown、文件和结构化数据库仍然是主数据。

---

## 7. 第三阶段：关系上下文与长期记忆

这一阶段扩展到：

- 人物档案
- 见面和沟通记录
- 关系图谱
- 重要承诺和待办
- 历史上下文回顾
- 联系人偏好和背景
- 事件时间线
- 个人决策记录

系统在用户再次联系某个人前，可以主动生成简洁背景卡片，但所有信息必须有来源和时间。

---

## 8. 第四阶段：消息过滤与个人助理

未来可接入经过授权的消息和通知渠道，提供：

- 消息重要性判断
- 群聊和通知摘要
- 待回复消息筛选
- 低价值信息降噪
- 回复草稿生成
- 重要事项提醒
- 决策上下文补充

关键原则：

- AI 默认不自动代表用户发送高风险消息
- 自动回复必须有明确边界和权限
- 财务、法律、医疗、合同和重要关系相关内容默认人工确认
- 所有代理行为可追踪、可撤销

---

## 9. 分阶段路线图

### Phase 0：定义与验证

- 明确边界和隐私原则
- 确定统一入口
- 确定开放数据格式
- 建立最小目录和字段结构

### Phase 1：收集箱

- 支持链接和文字输入
- AI 自动摘要、分类和标签
- 输出 Markdown
- NAS 自动备份
- 人工确认流程

### Phase 2：多媒体与自动解析

- 支持截图、PDF、语音和视频文本
- 自动提取实体和主题
- 去重和相似内容识别
- 来源可追溯

### Phase 3：检索与 RAG

- 语义检索
- 自然语言问答
- 返回原始引用
- 个性化搜索排序

### Phase 4：关系和事件记忆

- 人物档案
- 关系图谱
- 时间线
- 会议和沟通摘要

### Phase 5：助理与 Agent

- 消息过滤
- 任务建议
- 回复草稿
- 情境提醒
- 受控自动化

---

## 10. 当前结论

现阶段不需要先建设一个庞大的“人生操作系统”。

应该从一个可长期使用的个人知识助理入口开始：

```text
用户发送内容
  ↓
系统保存原文
  ↓
AI 摘要、分类、打标签
  ↓
用户确认
  ↓
Markdown + 数据库 + NAS 持久化
  ↓
未来接入检索、RAG、知识图谱和 Agent
```

第一阶段的成功标准不是功能多，而是用户看到有价值的信息时，愿意持续发送进来，并且以后能够真正找回来。
